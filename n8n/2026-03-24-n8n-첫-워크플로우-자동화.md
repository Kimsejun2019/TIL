### Context
n8n을 활용하여 첫 워크플로우 자동화를 구현했다. 이 워크플로우는 디스코드 메시지 수신을 트리거로 하여, LLM 에이전트를 통해 메시지를 TIL(Today I Learned) 형식의 기술 문서로 변환한다. 변환된 문서는 GitHub 레포지토리에 푸시되고, `README.md` 파일에 해당 문서의 인덱스가 자동으로 추가된다. 모든 과정이 완료되면 최종적으로 디스코드 채널에 완료 메시지를 발신한다.

### Core
n8n 워크플로우는 여러 노드(Node)들을 연결하여 데이터를 처리하고 특정 작업을 자동화한다. 본 워크플로우는 다음 노드들을 통해 일련의 자동화 과정을 수행한다:

*   **Discord Trigger 노드**: 특정 Discord 채널에서 새로운 메시지가 수신될 때 워크플로우를 시작하는 **시작 노드(Start Node)** 역할을 한다. 이 노드는 수신된 메시지의 내용, 발신자 정보 등을 포함하는 JSON 형식의 데이터를 다음 노드로 전달한다.
    *   **입력(Input)**: 없음 (워크플로우의 시작점)
    *   **출력(Output)**: Discord 메시지 데이터 (예: `{"message": {"content": "...", "author": "..."}}`)

*   **Code 노드 또는 HTTP Request 노드 (AI 에이전트 호출)**: Discord Trigger 노드로부터 받은 메시지 데이터를 AI 에이전트(LLM)에 전달하여 TIL 형식의 Markdown 문서를 생성하도록 요청한다. Code 노드를 사용하는 경우, Python/JavaScript 코드를 통해 직접 API 호출 로직을 구현할 수 있다. HTTP Request 노드를 사용하는 경우, 에이전트 API 엔드포인트에 HTTP POST 요청을 보낸다.
    *   **입력(Input)**: Discord 메시지 데이터
    *   **출력(Output)**: AI 에이전트가 생성한 TIL Markdown 텍스트 (예: `{"til_content": "# YYYY-MM-DD-제목\n### Context..."}`)

*   **GitHub 노드 (파일 생성 및 푸시)**: AI 에이전트가 생성한 TIL Markdown 텍스트를 GitHub 레포지토리에 새로운 파일로 커밋(commit)하고 푸시(push)한다. 파일명은 동적으로 생성되며, `YYYY-MM-DD-제목.md` 형식을 따른다.
    *   **입력(Input)**: TIL Markdown 텍스트 및 동적으로 생성된 파일명
    *   **출력(Output)**: GitHub 커밋/푸시 작업의 성공 여부 및 관련 정보

*   **GitHub 노드 (README.md 업데이트)**: 새로 추가된 TIL 문서의 링크를 레포지토리의 `README.md` 파일에 추가하여 문서 색인을 업데이트한다. 이는 `README.md`를 편집하고 다시 커밋하는 과정을 포함한다.
    *   **입력(Input)**: 새로 추가된 TIL 문서의 경로 및 `README.md` 파일의 현재 내용
    *   **출력(Output)**: `README.md` 업데이트 작업의 성공 여부

*   **Discord 노드 (완료 메시지 발신)**: 모든 GitHub 작업이 완료된 후, 사용자에게 자동화 과정이 성공적으로 끝났음을 알리는 완료 메시지를 Discord 채널에 발신한다.
    *   **입력(Input)**: 이전 노드들로부터 전달된 작업 완료 상태 또는 특정 완료 메시지 텍스트
    *   **출력(Output)**: Discord 메시지 발신 성공 여부

각 노드 간의 데이터 흐름(Data Flow)은 이전 노드의 **출력 데이터(Output Data)**가 다음 노드의 **입력 데이터(Input Data)**로 활용되는 방식으로 이루어진다. n8n은 내부적으로 JSON 형태의 데이터를 노드 간에 전달하며, 이를 통해 복잡한 자동화 로직을 구현할 수 있다.

### Insight
n8n을 활용하여 첫 워크플로우 자동화를 구축하면서 얻은 가장 중요한 통찰은 각 노드의 **입력(Input)**과 **출력(Output)** 특성, 그리고 이들 간의 **데이터 흐름(Data Flow)**을 명확히 이해하는 것이 워크플로우 설계의 핵심이라는 점이다. 특정 노드가 어떤 형식의 데이터를 입력으로 받아들이고, 어떤 구조의 데이터를 출력으로 내보내는지 파악하는 것은 효율적인 노드 연결과 예상치 못한 문제 해결에 필수적이다. 이는 n8n의 "Visual Workflow Automation" 철학과 일맥상통하며, 복잡한 로직을 시각적으로 구성하고 디버깅하는 데 큰 도움을 준다. 각 노드의 페이로드(Payload) 구조를 이해하는 것이 워크플로우의 성공적인 작동을 위한 기반이 된다.
