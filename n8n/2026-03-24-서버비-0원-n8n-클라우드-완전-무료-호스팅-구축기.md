### Context
본 문서는 n8n 워크플로우를 클라우드 환경에서 24시간 상시 운영하고자 하지만, 유료 서버 비용에 대한 부담을 느끼는 사용자를 위해 Google Cloud Platform(GCP)의 'Always Free' 정책을 활용하여 서버 비용 없이 n8n 인스턴스를 구축하는 전략을 다룹니다. GCP의 프리 티어(Free Tier) 인스턴스와 Docker를 이용한 환경 격리, 그리고 Ngrok을 통한 외부 접근성 확보 방안을 제시합니다.

### Core
이 섹션에서는 서버 비용 없이 n8n을 클라우드에 배포하기 위한 핵심 기술 스택 및 설정을 설명합니다.

*   **GCP e2-micro 인스턴스 활용**
    GCP의 'Always Free' 프로그램에 포함된 `e2-micro` 가상 머신 인스턴스를 활용하여 기본적인 컴퓨팅 자원을 확보합니다. `e2-micro` 인스턴스는 한 달 기준 특정 사용량까지 무료로 제공되어, 소규모 워크플로우에 적합한 환경을 제공합니다.

*   **Docker 및 `docker-compose`를 이용한 n8n 환경 구축**
    n8n 인스턴스를 컨테이너화하여 환경을 격리하고 관리를 용이하게 합니다. 다음은 `docker-compose.yml` 파일의 예시입니다.

    ```yml
    version: '3.8'

    services:
      n8n:
        image: n8n.io/n8n
        restart: always
        ports:
          - "5678:5678"
        environment:
          - N8N_HOST=localhost
          - N8N_PORT=5678
          - N8N_PROTOCOL=http
          - WEBHOOK_URL=http://<YOUR_NGROK_URL>
          - GENERIC_TIMEZONE=Asia/Seoul # 또는 원하는 타임존 설정
        volumes:
          - ~/.n8n:/home/node/.n8n
    ```
    *   `<YOUR_NGROK_URL>` 부분은 Ngrok 터널링 시 할당되는 외부 URL로 대체해야 합니다. `N8N_HOST`를 `0.0.0.0`으로 설정할 수도 있으나, Ngrok을 사용할 때는 `localhost`로 두어 내부 접근만 허용하고 Ngrok이 외부 트래픽을 처리하게 하는 것이 일반적입니다.

*   **Ngrok을 통한 외부 웹훅 노출**
    Ngrok은 고정 IP 설정이나 복잡한 방화벽 규칙 없이도 로컬에서 실행 중인 n8n 인스턴스에 대한 외부 접근 가능한 퍼블릭 URL을 제공합니다. GCP `e2-micro` 인스턴스 내에서 n8n이 실행되고 있는 포트(기본 5678)로 Ngrok 터널을 생성하여 외부 웹훅(Webhook) 요청을 수신할 수 있게 합니다.

    ```bash
    ngrok http 5678
    ```
    위 명령을 실행하면 `http://<random_subdomain>.ngrok.io` 또는 `https://<random_subdomain>.ngrok.io` 형식의 URL이 생성되며, 이를 `WEBHOOK_URL` 환경 변수에 설정하거나 n8n 워크플로우의 웹훅 노드에 입력하여 사용합니다.

### Insight
GCP Always Free와 Ngrok을 활용한 n8n 무료 호스팅은 초기 비용 없이 워크플로우를 테스트하고 운영할 수 있는 매력적인 방법입니다. 하지만 몇 가지 기술적 제약 사항을 고려해야 합니다.

*   **자원 한계**: GCP `e2-micro` 인스턴스는 CPU 2개(virtual CPUs), 1GB RAM의 제한된 자원을 제공합니다. 이는 복잡하거나 대량의 데이터를 처리하는 n8n 워크플로우에는 부적합할 수 있으며, 워크플로우 실행 중 성능 저하나 지연이 발생할 수 있습니다.
*   **Ngrok 무료 티어의 제약**: Ngrok 무료 티어는 터널이 재시작될 때마다 퍼블릭 URL이 변경되는 '임시 URL(Ephemeral URL)' 방식을 사용합니다. 이는 n8n 웹훅 URL을 수동으로 업데이트해야 하는 번거로움을 초래하며, 외부 시스템과의 연동 시 안정성을 저해할 수 있습니다. 또한, HTTP 요청 수(월 20,000건), TCP 연결 수(월 5,000건) 등 사용량에 제한이 있어 고빈도 또는 대규모 웹훅 처리가 필요한 경우에는 적합하지 않습니다. HTTPS가 아닌 HTTP 연결만 지원하며, 사용자 지정 도메인(Custom Domain) 및 TLS/SSL 인증서 지원도 무료 티어에서는 제한됩니다.
*   **지속적인 운영의 어려움**: Ngrok의 임시 URL 문제와 GCP 인스턴스의 안정성(간헐적 재부팅 등)으로 인해, 24시간 무중단 운영이 필수적인 프로덕션 환경에서는 이 구성이 적합하지 않을 수 있습니다. 개인 프로젝트나 학습용으로는 훌륭한 대안이 될 수 있습니다.
